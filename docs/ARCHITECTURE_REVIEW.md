## 一、项目现状评估（基于代码的真实评估）

### 1.1 当前架构图（实际运行状态）

```
用户输入："帮我写一份关于安全检查的通知"
    │
    ▼
┌─────────────────────────────────────────┐
│  前端 ChatPage.tsx                       │
│  → 调用 /api/chat/messages/stream       │
└──────────────────┬──────────────────────┘
                   │ POST (SSE流式)
                   ▼
┌─────────────────────────────────────────┐
│  后端 chat.ts                            │
│                                          │
│  systemPrompt = "你是政府公文写作专家"    │  ← 固定字符串！
│  messages = [system, ...历史消息]         │
│                                          │
│  → 直接调用 DeepSeek API                 │  ← 没有任何中间处理！
│  → 流式返回结果                           │
└─────────────────────────────────────────┘
```

**结论：当前系统本质上是 "固定Prompt + DeepSeek API" 的薄封装。**

### 1.2 已实现 vs 未实现

| 模块 | 代码存在？ | 实际工作？ | 评价 |
|:---|:---:|:---:|:---|
| 用户认证 | ✅ | ✅ | JWT + SQLite，可用 |
| 对话管理 | ✅ | ✅ | CRUD完整，SSE流式 |
| 公文生成 | ✅ | ✅ | DeepSeek直连，可用 |
| Agent写作 | ✅ | ⚠️ | LangGraph框架搭好，6步流程只有骨架 |
| 语料库存储 | ✅ | ❌ | Prisma模型存在，但依赖PostgreSQL未配置 |
| 向量检索 | ❌ | ❌ | agent.ts中有占位代码，无实际实现 |
| 记忆系统 | ❌ | ❌ | agent.ts写了"MVP阶段使用默认记忆" |
| 风格学习 | ❌ | ❌ | 完全不存在 |
| RAG增强 | ❌ | ❌ | 无检索、无注入、无增强 |

### 1.3 致命缺陷

**1. 系统提示词是硬编码的死字符串**
```typescript
// chat.ts 第143行 — 这就是全部的"智能"
const systemPrompt = conversation.type === 'speech' 
  ? '你是一位资深的政府公文写作专家，专门协助用户撰写讲话稿。'
  : '你是一位资深的政府公文写作专家。';
```
不管用户是谁、写过什么、喜欢什么风格——系统永远说同一句话。

**2. 两套数据库互不相通**
- `mockDB.ts`（SQLite）→ 认证、对话、消息
- `Prisma`（PostgreSQL）→ 语料库
- 结果：语料库和对话系统完全隔离，无法互相引用

**3. Agent框架是空壳**
- `agent.ts` 定义了6个节点（分析→记忆→检索→大纲→生成→润色）
- 但记忆节点返回空对象，检索节点只是占位
- 而且这个Agent**根本没有接入聊天界面**

## 二、核心问题诊断

### 问题本质

> **用户期望：** "越用越懂我"  
> **系统现实：** "每次都是第一次见面"

这不是一个bug，是**架构缺失**。当前系统没有任何机制来：
1. 记住用户说过什么
2. 分析用户的写作偏好
3. 在生成时利用这些知识

### 用一个比喻来理解

想象你雇了一个秘书：

| 阶段 | 秘书的表现 | 对应系统能力 |
|:---|:---|:---|
| **第1天** | 按标准模板写公文 | 当前系统（固定Prompt） |
| **第1周** | 记住你喜欢简洁风格 | 短期记忆（需实现） |
| **第1月** | 了解你的用语习惯、常用结构 | 长期记忆（需实现） |
| **第3月** | 能模仿你的写作风格 | 风格模型（需实现） |
| **第6月** | 不用你说就知道怎么写 | 预测性生成（远期目标） |

**当前系统永远停在"第1天"。**

## 三、记忆系统深度设计（核心章节）

### 3.1 什么是"记忆"？

记忆不是一个功能，是**四层能力的叠加**：

```
┌─────────────────────────────────────────────┐
│  第4层：语义记忆（知识库）                    │
│  "安全检查通知一般包含：检查范围、时间、要求"  │
│  → 来源：语料库中的范文和知识                  │
│  → 存储：向量数据库                           │
│  → 寿命：永久                                 │
├─────────────────────────────────────────────┤
│  第3层：长期记忆（用户画像）                   │
│  "这个用户偏好简洁风格，常用四字成语，         │
│   习惯用'着力'、'切实'等词汇"                 │
│  → 来源：从历史对话中提取                      │
│  → 存储：结构化JSON + PostgreSQL              │
│  → 寿命：持久，缓慢更新                       │
├─────────────────────────────────────────────┤
│  第2层：短期记忆（近期偏好）                   │
│  "最近三次都要求1500字以内"                    │
│  "上次让我把'进一步'改成了'持续'"              │
│  → 来源：近期对话                             │
│  → 存储：会话缓存                             │
│  → 寿命：7-30天，逐渐衰减                     │
├─────────────────────────────────────────────┤
│  第1层：工作记忆（当前对话）                   │
│  "这次要写安全检查通知，要求正式语气"          │
│  → 来源：当前对话上下文                        │
│  → 存储：LLM上下文窗口                        │
│  → 寿命：当前会话                             │
└─────────────────────────────────────────────┘
```

### 3.2 如何从用户行为中提取风格特征？

**关键洞察：用户的"修改"比用户的"要求"更能反映真实偏好。**

```
场景：用户说"帮我写一份通知"
    │
    ▼
AI生成了一篇2000字的通知
    │
    ▼
用户说："太长了，缩短到800字，'进一步加强'改成'切实做好'"
    │
    ▼
    ┌──────────────────────────────────┐
    │  风格提取引擎（LLM分析）          │
    │                                   │
    │  输入：原文 + 用户修改指令         │
    │  提取：                           │
    │   - 偏好长度：800字（简短）        │
    │   - 词汇偏好：切实 > 进一步       │
    │   - 风格标签：简洁、务实           │
    │                                   │
    │  输出：风格特征更新                │
    └──────────────────────────────────┘
    │
    ▼
存入用户风格档案（UserStyleProfile）
```

**具体实现方案：**

```typescript
// 用户风格档案数据结构
interface UserStyleProfile {
  userId: string;
  
  // 基础偏好
  preferredLength: "short" | "medium" | "long";  // 偏好篇幅
  formalityLevel: number;     // 正式程度 1-10
  
  // 词汇偏好（从修改中学习）
  preferredTerms: {           // 用户偏好的用词
    term: string;             // "切实"
    replacedFrom: string;     // 替代了"进一步"
    frequency: number;        // 出现频次
    lastUsed: Date;
  }[];
  
  // 结构偏好
  preferredStructure: {
    usesNumberedList: boolean;    // 是否偏好数字编号
    paragraphStyle: "short" | "long";  // 段落长度偏好
    usesSubheadings: boolean;     // 是否使用小标题
  };
  
  // 语气特征
  toneMarkers: string[];      // ["务实", "简洁", "数据驱动"]
  
  // 元数据
  totalInteractions: number;   // 总交互次数
  lastUpdated: Date;
  confidence: number;          // 画像置信度 0-1
}
```

│  → 3篇相关文档    档案          → 2篇范文   │

┌─ ③ 提示词组装 ────────────────────────────┐
│                                            │

│  你是政府公文写作专家。                     │
│                                            │
│  [用户风格档案]  ← 第3层：长期记忆          │

│                                            │
│  [近期偏好]  ← 第2层：短期记忆              │

│  "把开头的套话删掉，直接进入正题"            │
│                                            │

                   │
                   ▼

                   │
                   ▼
┌─ ④ 后处理 ────────────────────────────────┐
│  - 检查是否符合用户风格偏好                 │
│  - 如果用户后续修改 → 更新风格档案          │
└────────────────────────────────────────────┘
```

### 3.4 记忆的衰减、更新、冲突解决

**衰减机制：**
```
记忆权重 = 基础权重 × 时间衰减因子 × 频次加成

时间衰减因子 = e^(-λt)
  λ = 0.1（短期记忆，7天半衰期）
  λ = 0.01（长期记忆，70天半衰期）
  λ = 0（语义记忆，不衰减）
```

**更新机制：**
- 每次对话结束 → 异步提取风格特征 → 与现有档案合并
- 新偏好权重 > 旧偏好 → 更新
- 矛盾偏好 → 以最近3次为准

**冲突解决：**
```
场景：用户上周要求"2000字"，这周要求"简短"
    │
    ▼
规则：显式指令 > 隐式偏好 > 历史默认
    │
    ▼
结果：本次按"简短"生成，同时更新偏好为"偏好简短"
```

│      ▼                                   │
│  注入LLM上下文                            │
│  "参考以下范文片段生成，保持类似的        │
│   结构和用语风格：[片段1] [片段2]..."      │
└─────────────────────────────────────────┘
```

### 4.2 关键技术选择

| 组件 | 推荐方案 | 理由 |
|:---|:---|:---|
| **向量数据库** | pgvector（PostgreSQL扩展） | 已有PostgreSQL，无需新增服务 |
| **Embedding模型** | OpenAI text-embedding-3-small | 性价比最高，$0.02/1M tokens |
| **分块策略** | 按段落 + 重叠滑窗 | 保持语义完整性 |
| **检索策略** | 混合检索（向量+关键词） | 兼顾语义和精确匹配 |

## 六、分阶段实施路线图

### Phase 4a：统一数据库 + RAG基础（1周）

```
□ 将mockDB迁移到Prisma/PostgreSQL（统一数据库）

**里程碑：用过3-5次后，系统开始了解用户的写作偏好。**

□ 用户修改行为追踪（diff分析）

```

**里程碑：用户说"按我的风格写"，系统真的能按用户风格生成。**

## 七、风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|:---|:---:|:---:|:---|
| pgvector性能不足 | 低 | 中 | 语料量小（<1万条），pgvector足够 |
| Embedding成本超预期 | 低 | 低 | text-embedding-3-small极便宜 |
| 风格提取不准确 | 中 | 高 | 初期用LLM提取，后期可训练专用模型 |
| 用户感知不到"学习" | 中 | 高 | 5次交互后主动提示"已学习到您的偏好" |
| Render免费计划性能 | 高 | 中 | 考虑升级$7/月付费计划 |

## 八、给老板的总结

**当前系统 vs 目标系统：**

| 维度 | 当前 | 目标 |
|:---|:---|:---|
| **理解能力** | 不理解用户 | 越用越懂 |
| **语料利用** | 完全不用 | 参考范文生成 |
| **风格学习** | 零 | 自动提取+应用 |
| **记忆持久** | 每次重新开始 | 跨会话持久记忆 |
| **改造工作量** | - | **约3周** |

**最关键的一步：把固定的系统提示词变成动态组装的提示词。** 这一步改完，后续所有记忆和RAG能力都能接入。
