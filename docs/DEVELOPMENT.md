# WXD 项目开发流程规范

**文档版本**: v1.1  
**生效日期**: 2026-03-28  
**适用范围**: WXD 写作助手项目全体成员  
**文档状态**: 强制执行  
**设计原则**: 流程是手段，不是目的。精简高效，拒绝过度工程。

---

## 一、流程总览

```
需求确认 → 技术对齐 → 开发 → 测试 → 部署 → 监控
   │           │         │       │       │       │
 飞书任务    口头/文档   PR+Review 分层测试  自动化   Sentry
 1页AC      复杂才写    24h响应  核心覆盖  Preview  +Analytics
```

**核心理念:** 简单需求快速流转，复杂需求才走完整流程。

---

## 二、需求确认阶段

### 2.1 需求分级

| 级别 | 定义 | 流程 |
|------|------|------|
| **S-小** | ≤ 半天，Bug修复/文案调整 | 飞书任务 → 直接开发 |
| **M-中** | 1-2天，独立功能模块 | 飞书任务 + 简要AC → 开发 |
| **L-大** | ≥ 3天，跨模块/架构变更 | 完整评审流程 |

### 2.2 飞书任务单（所有需求必填）

```
标题: [动词开头] 简明描述
描述:
  - 背景: 为什么做？
  - 目标: 做成什么样？
  - AC: 验收标准（可测试的条件列表）
标签: 优先级(P0/P1/P2) + 类型(feature/bug/refactor)
负责人: 唯一
截止日期: 明确
```

### 2.3 完整评审（仅L级需求）

**评审 Checklist:**
- [ ] 用户场景是否完整（谁、什么情况、做什么）？
- [ ] 异常流程和边界情况是否考虑？
- [ ] 是否符合现有技术架构？
- [ ] 是否涉及数据模型变更？
- [ ] 性能与安全影响评估？

**评审结论:**

| 结论 | 含义 | 后续 |
|------|------|------|
| **通过** | 进入开发 | 创建技术方案 |
| **有条件通过** | 小修改 | 飞书确认即可 |
| **不通过** | 重大问题 | 重新准备 |

## 三、技术对齐阶段

### 3.1 按需求级别选择深度

**S/M级需求:** 口头对齐或飞书任务描述中说明即可。

**L级需求:** 产出 `docs/tech/{feature-name}.md`，包含：

```markdown
# XX功能技术方案

## 1. 需求概述
- 背景、目标、范围边界

## 2. 架构设计
- 模块划分、数据流、接口定义
- API 请求/响应示例
- 数据库变更（如有）

## 3. 风险与回滚
- 技术风险与缓解措施
- 回滚方案
- 测试策略
```

### 3.2 架构决策记录（ADR）

涉及技术选型、架构调整时，必须记录 `docs/decisions/ADR-xxx.md`：

```markdown
# ADR-XXX: [决策标题]

**状态**: 已采纳 / 已废弃
**日期**: YYYY-MM-DD
**决策者**: @who

## 背景
为什么需要做这个决策？

## 决策
选择了什么？

## 理由
为什么选这个？对比了什么替代方案？

## 后果
正面影响、负面影响、技术债务
```

## 四、开发阶段

### 4.1 分支管理（Trunk-Based）

```
main (生产分支，唯一长期分支)
  ↑
feature/xxx (短生命周期，≤ 3天合并)
  ↑
hotfix/xxx (紧急修复，当天合并)
```

**规则:**
- **不使用 develop 分支**，feature 直接合入 main
- Feature 分支生命周期不超过 3 天
- 合并方式：Squash Merge
- 合并前必须通过 CI + Code Review

**分支命名:**
```
feature/add-password-reset      # 功能
bugfix/fix-corpus-upload        # 缺陷
hotfix/critical-auth-bypass     # 紧急
```

### 4.2 提交规范

**格式:**
```
<type>(<scope>): <subject>
```

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | `feat(auth): add password reset` |
| fix | 缺陷修复 | `fix(ui): button alignment` |
| docs | 文档 | `docs(api): update swagger` |
| refactor | 重构 | `refactor(db): optimize query` |
| test | 测试 | `test(api): add unit tests` |
| chore | 构建/工具 | `chore: update dependencies` |

**Subject 规范:** 动词开头，≤50字符，不加句号。

### 4.3 Code Review

**流程:**
```
开发完成 → 提交PR → Reviewer 24小时内响应 → 修改 → Squash Merge
```

**Reviewer 分配:**

| 变更类型 | Reviewer |
|----------|----------|
| 前端代码 | @frontend-lead |
| 后端代码 | @backend-lead |
| 数据库/架构变更 | @backend-lead + @KP |
| 部署配置 | @devops-lead |

**Review Checklist:**

*必查:*
- [ ] 无 console.log / debugger 残留
- [ ] 无敏感信息硬编码（密钥、密码、Token）
- [ ] 错误处理完善
- [ ] 输入参数有校验

*前端额外:*
- [ ] TypeScript 类型完整
- [ ] 响应式设计

*后端额外:*
- [ ] API 符合 RESTful 规范
- [ ] 数据库事务正确
- [ ] 日志记录完善

**结论:** `LGTM`(合并) / `Request Changes`(修改) / `Comment`(仅评论)

## 五、测试阶段

### 5.1 分层测试标准

| 测试类型 | 工具 | 覆盖要求 | 执行时机 |
|----------|------|----------|----------|
| 单元测试 | Jest/Vitest | 核心业务 ≥70%，工具函数 ≥90%，UI ≥30% | 每次提交 |
| 集成测试 | Supertest | 核心 API 流程覆盖 | PR 合并前 |
| E2E测试 | Playwright | 主流程 Happy Path | 发布前 |
| 安全扫描 | ESLint Security | 无高危漏洞 | PR 合并前 |

### 5.2 测试 Checklist

**每次PR必须:**
- [ ] 新增代码有对应测试
- [ ] 修改代码的已有测试仍通过
- [ ] 边界条件和异常路径覆盖

**发布前必须:**
- [ ] E2E 主流程通过
- [ ] 无已知 P0/P1 缺陷

## 六、部署流程

### 6.1 环境划分

| 环境 | 用途 | 部署触发 | 数据库 |
|------|------|----------|--------|
| Local | 本地开发 | 手动 | SQLite |
| Preview | PR预览/集成验证 | 自动 (Vercel/Render) | 共享测试库 |
| Production | 生产 | 合并到 main 自动触发 | 生产库 (PostgreSQL) |

### 6.2 部署流程

```
PR合并到main → CI自动构建 → 自动部署 → 部署后验证
      │              │             │            │
    Squash        Vercel/       Preview→Prod   检查单
    Merge         Render                       验收
```

**Frontend (Vercel):** PR 自动 Preview，合并 main 自动 Production  
**Backend (Render):** 合并 main 自动部署，重大变更手动触发

### 6.3 部署检查单

**部署前:**
- [ ] 所有测试通过
- [ ] Code Review 完成
- [ ] 数据库迁移脚本已验证（如有）
- [ ] 回滚方案已确认

**部署后:**
- [ ] 健康检查 `/health` 通过
- [ ] 核心功能手动验证
- [ ] 监控无异常告警

## 七、监控与运维

### 7.1 监控工具

| 类别 | 工具 | 监控内容 |
|------|------|----------|
| 前端性能 | Vercel Analytics | 页面加载、Core Web Vitals |
| 错误追踪 | Sentry | 运行时异常、堆栈跟踪 |
| 后端日志 | Render Logs | 应用日志、请求日志 |
| 告警通知 | 飞书机器人 (卡维斯) | 异常推送、每日汇报 |

### 7.2 告警策略

| 级别 | 条件 | 通知方式 | 响应时间 |
|------|------|----------|----------|
| **P0** | 服务不可用 | 飞书即时推送 | 15分钟 |
| **P1** | 核心功能异常 | 飞书推送 | 1小时 |
| **P2** | 错误率升高 | 飞书推送 | 4小时 |
| **P3** | 预警/优化建议 | 每日汇报汇总 | 24小时 |

### 7.3 例行检查

**每日 (卡维斯自动执行):**
- 错误率、响应时间、数据库连接

**每周 (周五回顾时覆盖):**
- 性能趋势、错误分布、资源使用率

**每月:**
- 成本分析、安全审计、文档更新

## 八、文档管理

### 8.1 目录结构

```
docs/
├── ARCHITECTURE_REVIEW.md    # 架构设计
├── DEVELOPMENT.md            # 开发规范 (本文件)
├── CHANGELOG.md              # 变更日志
├── requirements/             # 需求文档 (L级需求)
├── tech/                     # 技术方案 (L级需求)
├── decisions/                # 架构决策记录 (ADR)
└── runbooks/                 # 运维手册
```

### 8.2 文档更新触发

| 事件 | 更新 |
|------|------|
| API变更 | CHANGELOG |
| 架构调整 | ADR + ARCHITECTURE |
| 新功能上线 | CHANGELOG |
| 生产故障 | runbooks/ |

## 九、沟通协作

### 9.1 会议机制

| 会议 | 频率 | 时长 | 内容 |
|------|------|------|------|
| 迭代规划 | 周一 | 30分钟 | 本周目标、任务分配 |
| 周五回顾 | 周五 | 30分钟 | 本周成果、问题复盘、下周计划 |
| 技术评审 | 按需 (L级) | 45分钟 | 架构设计审查 |

**日常沟通:** 飞书异步，卡维斯每日 8:30 自动汇报任务进展。

### 9.2 飞书任务状态流转

```
待开发 → 开发中 → 待Review → 待部署 → 已完成
```

## 十、发布合规检查单

每次生产发布前，卡维斯确认：

- [ ] 飞书任务AC全部满足
- [ ] Code Review 已完成 (LGTM)
- [ ] 测试通过（CI绿灯）
- [ ] 数据库变更已验证（如有）
- [ ] 部署后验证通过
- [ ] CHANGELOG 已更新
