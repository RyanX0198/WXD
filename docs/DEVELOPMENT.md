## 二、需求评审阶段

### 2.1 准入标准
需求必须满足以下全部条件方可进入开发流程：

| 检查项 | 标准 | 责任人 |
|--------|------|--------|
| 需求文档 | PRD 文档已创建并存放于 `docs/requirements/` | 产品经理 |
| 用户故事 | 包含清晰的用户故事格式 (作为...我想要...以便...) | 产品经理 |
| 验收标准 | 定义明确的 AC (Acceptance Criteria) | 产品经理 |
| 原型设计 | Figma/Sketch 原型链接已提供 | 设计师 |
| 依赖识别 | 外部依赖项已列出并评估风险 | 技术负责人 |

### 2.2 需求评审 Checklist

**逻辑完整性检查:**
- [ ] 需求背景和业务目标是否清晰？
- [ ] 用户场景描述是否完整（谁、在什么情况下、做什么）？
- [ ] 异常流程和边界情况是否考虑？
- [ ] 与其他模块的交互关系是否明确？
- [ ] 数据流向是否清晰（输入、处理、输出）？

**架构合理性检查:**
- [ ] 该需求是否符合现有技术架构？
- [ ] 是否需要引入新技术/新服务？
- [ ] 是否涉及数据模型变更？
- [ ] 性能影响评估（并发、响应时间）
- [ ] 安全性考虑（权限、数据隔离）

**文档齐备性检查:**
- [ ] PRD 文档格式是否符合模板？
- [ ] API 接口文档是否已定义（如需要）？
- [ ] 数据库变更文档（如需要）？
- [ ] 测试用例是否已编写？

### 2.3 评审会议流程

```
1. 会前 (T-1天)
   └─ 评审材料提前24小时发出
   └─ 与会者预先阅读并准备问题

2. 会中 (30-60分钟)
   └─ 需求提出者讲解 (10分钟)
   └─ 技术可行性讨论 (20分钟)
   └─ 风险识别与应对方案 (15分钟)
   └─ 明确责任人与时间节点 (5分钟)

3. 会后 (T+0天)
   └─ 会议纪要发送全员
   └─ 评审结论录入飞书任务系统
   └─ 变更需求更新 PRD
```

### 2.4 评审结论

| 结论 | 含义 | 后续动作 |
|------|------|----------|
| **通过** | 可进入设计阶段 | 创建技术方案文档 |
| **有条件通过** | 需小修改 | 修改后邮件确认，无需二次会议 |
| **不通过** | 需重大修改 | 重新准备，安排二次评审 |
| **暂缓** | 优先级或资源问题 | 放入 Backlog，说明触发条件 |

## 三、技术方案评审阶段

### 3.1 技术方案文档模板

每个需求必须产出 `docs/tech/{feature-name}.md`，包含：

```markdown
# XX功能技术方案

## 1. 需求概述
- 需求背景
- 预期目标
- 范围边界

## 2. 架构设计
- 系统架构图
- 模块划分
- 数据流图

## 3. 接口设计
- API 定义 (OpenAPI/Swagger 格式)
- 请求/响应示例
- 错误码定义

## 4. 数据库设计
- ER 图
- 表结构变更
- 索引策略
- 迁移脚本

## 5. 技术选型
- 技术栈选择理由
- 第三方服务评估
- 替代方案对比

## 6. 风险与应对
- 技术风险
- 缓解措施
- 回滚方案

## 7. 测试策略
- 单元测试覆盖率目标
- 集成测试范围
- 性能测试指标

## 8. 部署计划
- 环境要求
- 部署步骤
- 配置变更
```

### 3.2 技术评审 Checklist

**架构合理性:**
- [ ] 是否遵循单一职责原则？
- [ ] 耦合度是否可控？
- [ ] 是否可水平扩展？
- [ ] 是否符合现有代码规范？
- [ ] 是否考虑了向后兼容性？

**技术选型:**
- [ ] 新技术是否经过 POC 验证？
- [ ] 开源组件许可证是否合规？
- [ ] 社区活跃度是否满足要求？
- [ ] 学习成本是否可控？

**资源配置:**
- [ ] 开发人力评估是否准确？
- [ ] 是否需要额外基础设施？
- [ ] 外部服务成本是否评估？
- [ ] 测试环境资源是否到位？

### 3.3 资源申请流程

```
开发资源申请 → 技术负责人审批 → DevOps分配 → 确认到位
     │                │              │           │
   填写飞书        评估合理性      创建环境      验收确认
   任务单          优先级排序      分配权限      更新状态
```

**资源类型清单:**
- [ ] 开发环境 (本地/远程)
- [ ] 测试环境 (Vercel Preview/Render Staging)
- [ ] 数据库实例
- [ ] 第三方服务账号
- [ ] 设计资源 (Figma/Sketch)

## 四、开发阶段

### 4.1 分支管理策略

采用 **Git Flow** 简化版：

```
main (生产分支)
  ↑
release/v1.x (预发布分支，可选)
  ↑
develop (开发主分支)
  ↑
feature/xxx (功能分支)
  ↑
hotfix/xxx (热修分支)
```

**分支命名规范:**
```
feature/PROJ-123-add-login-page     # 功能分支
bugfix/PROJ-456-fix-memory-leak     # 缺陷修复
hotfix/PROJ-789-critical-security   # 线上热修
release/v1.2.0                      # 发布分支
```

### 4.2 提交规范

**Commit Message 格式:**
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 定义:**
| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | `feat(auth): add password reset` |
| fix | 缺陷修复 | `fix(ui): button alignment` |
| docs | 文档更新 | `docs(api): update swagger` |
| style | 代码格式 | `style: remove unused imports` |
| refactor | 重构 | `refactor(db): optimize query` |
| test | 测试相关 | `test(api): add unit tests` |
| chore | 构建/工具 | `chore: update dependencies` |

**Subject 规范:**
- 不超过 50 个字符
- 使用动词开头 (add/fix/update/remove)
- 不使用句号结尾

### 4.3 代码管理规范

**开发前:**
- [ ] 从 develop 创建 feature 分支
- [ ] 在飞书任务系统中关联分支
- [ ] 更新任务状态为「开发中」

**开发中:**
- [ ] 每个工作日至少提交一次代码
- [ ] 提交前执行本地 lint 检查
- [ ] 提交信息遵循规范

**开发完成:**
- [ ] 本地测试通过
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 自测 checklist 完成
- [ ] 更新 CHANGELOG

### 4.4 Code Review 规范

**Review 流程:**
```
开发者提交PR → 指定 Reviewer → Review → 修改 → 合并
     │              │              │        │        │
   填写PR        至少1人        48小时内   逐条     Squash
   描述          Frontend/      响应      回复     Merge
                 Backend
```

**Reviewer 分配:**
| 变更类型 | 必需 Reviewer |
|----------|--------------|
| 前端代码 | @frontend-lead |
| 后端代码 | @backend-lead |
| 数据库变更 | @backend-lead + @KP |
| 架构调整 | @KP |
| 部署配置 | @devops-lead |

**Review Checklist:**

*通用:*
- [ ] 代码符合项目规范
- [ ] 无 console.log / debugger 残留
- [ ] 无敏感信息硬编码
- [ ] 错误处理完善

*前端:*
- [ ] TypeScript 类型定义完整
- [ ] 组件 Props 有 JSDoc
- [ ] 响应式设计考虑
- [ ] 无障碍属性 (a11y)

*后端:*
- [ ] API 遵循 RESTful 规范
- [ ] 输入参数有校验
- [ ] 数据库事务处理正确
- [ ] 日志记录完善

**Review 结论标签:**
- `LGTM` - 看起来不错，可以合并
- `Request Changes` - 需要修改
- `Comment` - 仅评论，无阻塞问题

## 五、测试阶段

### 5.1 测试金字塔

```
      /\
     /  \
    / E2E \      ← 端到端测试 (少量)
   /--------\
  / Integration\  ← 集成测试 (中等)
 /--------------\
/    Unit Tests  \ ← 单元测试 (大量)
------------------
```

### 5.2 测试类型与标准

| 测试类型 | 工具 | 覆盖率要求 | 执行时机 |
|----------|------|------------|----------|
| 单元测试 | Jest/Vitest | ≥ 80% | 每次提交 |
| 集成测试 | Jest + Supertest | 核心流程覆盖 | PR合并前 |
| E2E测试 | Playwright | 主流程 | 发布前 |
| 性能测试 | k6/Artillery | 基线对比 | 重大变更 |
| 安全扫描 | ESLint Security | 无高危漏洞 | PR合并前 |

### 5.3 测试 checklist

**单元测试:**
- [ ] 每个函数/方法至少一个测试
- [ ] 边界条件覆盖
- [ ] 异常路径测试
- [ ] Mock 外部依赖

**集成测试:**
- [ ] API 端到端测试
- [ ] 数据库交互测试
- [ ] 第三方服务 Mock

**E2E测试:**
- [ ] 用户主流程 Happy Path
- [ ] 关键错误场景
- [ ] 跨浏览器兼容性

## 六、部署流程

### 6.1 环境划分

| 环境 | 用途 | 部署触发 | 数据库 |
|------|------|----------|--------|
| Local | 本地开发 | 手动 | SQLite |
| Preview | PR预览 | 自动(Vercel) | 共享测试库 |
| Staging | 集成测试 | 手动/定时 | 独立测试库 |
| Production | 生产 | 手动审批 | 生产库 |

### 6.2 部署检查单

**部署前:**
- [ ] 所有测试通过
- [ ] Code Review 完成
- [ ] 数据库迁移脚本已验证
- [ ] 配置文件已更新
- [ ] 回滚方案已确认

**部署中:**
- [ ] 部署窗口期确认
- [ ] 监控Dashboard打开
- [ ] 日志流监控

**部署后:**
- [ ] 健康检查通过
- [ ] 核心功能验证
- [ ] 监控指标正常
- [ ] 通知相关方

### 6.3 自动化部署流程

```
代码合并 → CI构建 → 测试 → 部署预览 → 人工确认 → 生产部署
   │          │       │        │          │         │
  GitHub   Vercel/  Jest    Preview    检查单    Render
  Actions  Render   测试    环境       审批      生产
```

**Frontend (Vercel):**
- 每次 PR 自动生成 Preview 链接
- 合并到 main 自动部署 Production

**Backend (Render):**
- 每次 PR 自动生成 Preview 服务
- 生产部署需手动触发

## 七、监控与运维

### 7.1 监控体系

```
┌─────────────────────────────────────────────────────────────┐
│                      监控Dashboard                           │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │  性能指标   │  │  错误监控   │  │  业务指标   │         │
│  │             │  │             │  │             │         │
│  │ • 响应时间  │  │ • 错误率    │  │ • DAU/MAU   │         │
│  │ • 吞吐量    │  │ • 异常类型  │  │ • 转化率    │         │
│  │ • 资源使用  │  │ • 堆栈跟踪  │  │ • 留存率    │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 监控工具

| 类别 | 工具 | 监控内容 |
|------|------|----------|
| 性能 | Vercel Analytics | 页面加载、Core Web Vitals |
| 错误 | Sentry | 运行时错误、性能问题 |
| 日志 | Render Logs | 应用日志、访问日志 |
| 告警 | 飞书机器人 | 异常通知、每日报告 |

### 7.3 告警策略

| 级别 | 条件 | 通知方式 | 响应时间 |
|------|------|----------|----------|
| P0-紧急 | 服务不可用/严重错误 | 飞书+短信 | 15分钟 |
| P1-高 | 性能严重下降 | 飞书 | 1小时 |
| P2-中 | 错误率升高 | 飞书 | 4小时 |
| P3-低 | 预警信息 | 日报汇总 | 24小时 |

### 7.4 监控检查单

**每日检查:**
- [ ] 错误率是否异常？
- [ ] 响应时间是否达标？
- [ ] 数据库连接是否正常？
- [ ] 磁盘空间是否充足？

**每周检查:**
- [ ] 性能趋势分析
- [ ] 错误分布分析
- [ ] 用户行为分析
- [ ] 资源使用率评估

**每月检查:**
- [ ] 容量规划评估
- [ ] 成本分析
- [ ] 安全审计
- [ ] 文档更新

## 九、沟通协作

### 9.1 飞书任务管理

**任务状态流转:**
```
待评审 → 待开发 → 开发中 → 待测试 → 待部署 → 已完成
          ↑________↓
            开发中
```

**任务字段要求:**
- [ ] 标题：清晰的动词开头
- [ ] 描述：背景、目标、验收标准
- [ ] 标签：优先级(P0/P1/P2)、类型(feature/bug)
- [ ] 负责人：唯一责任人
- [ ] 截止日期：明确的时间点
- [ ] 关联：关联PR、关联文档

### 9.2 会议机制

| 会议 | 频率 | 参与人 | 时长 |
|------|------|--------|------|
| 每日站会 | 每日 | 全员 | 15分钟 |
| 需求评审 | 按需 | 产品+技术 | 60分钟 |
| 技术评审 | 按需 | 技术团队 | 45分钟 |
| 迭代规划 | 每周 | 全员 | 60分钟 |
| 回顾会议 | 每两周 | 全员 | 45分钟 |

### 9.3 汇报机制

**每日工作汇报 (8:30):**
- 昨日完成
- 今日计划
- 阻塞问题

**每周总结 (周五):**
- 本周里程碑
- 下周计划
- 风险预警
